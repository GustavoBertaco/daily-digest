## AI Engineer

**[In Code They Act, In Proof We Trust — Erik Meijer, Leibniz Labs](https://www.youtube.com/watch?v=-CnA2lGfymY)**
Erik Meijer's thesis: AI agents acting on blind trust produce predictable, not exceptional, failures — and the already-documented incidents prove that allowing agents to act without mechanical correctness guarantees is an architectural flaw, not a tuning problem.

- **The failures are not edge cases.** A dealership chatbot selling a $76,000 Chevy Tahoe for $1, a coding agent wiping a production database during a code freeze, an agent skill quietly installing a keylogger — these are cited as direct, predictable consequences of execution without verification.
- **Execution is irreversible.** The snippet states this explicitly: you cannot unsend a message, undelete a database row, or undo a file overwrite. The asymmetry between the cost of a wrong action and the ease of triggering one is the core structural risk.
- **Formal proof as the proposed remedy.** The title — "In Code They Act, In Proof We Trust" — signals that the answer is mechanically verifiable correctness before an agent acts, not better prompts or post-hoc monitoring.

**[Stop Evaluating Models Like It's the 50s - Alejandro Vidal, Mindmakers](https://www.youtube.com/watch?v=O3FEoMYvUf8)**
Vidal's argument: AI evaluation runs on Classical Test Theory — count right answers, treat all questions as equal, trust the percentage — while psychology has spent a century developing superior measurement science (Item Response Theory) for exactly this kind of problem, and the AI field has not borrowed it.

- **Classical Test Theory is the wrong tool.** Summing raw scores treats every benchmark item as equally informative and equally difficult, which it is not — making leaderboard percentages an unreliable signal of actual model capability.
- **Item Response Theory (IRT) is the alternative.** The math behind the SAT and GRE models each question's difficulty and discriminating power separately, then infers a latent ability estimate rather than a raw score — a more robust and stable measure.
- **The analogy to psychology is direct.** Psychologists spent the last century measuring something "invisible and uncooperative" (the human mind); the parallel to AI capability measurement is structural and largely unexploited.

**[From fork() to Fleet: Designing an Agent Sandbox Cloud — Abhishek Bhardwaj, OpenAI](https://www.youtube.com/watch?v=OqM67QG_Ikk)**
OpenAI's Abhishek Bhardwaj covers the architectural decisions behind building a cloud of agent sandboxes — where runtime isolation technology choice, storage persistence, and fleet orchestration are the three central engineering problems.

- **Sandboxes unlock agent capability.** Secure, fully functional compute environments let agents tackle diverse tasks with minimal setup; the sandbox is not merely a safety layer but the infrastructure that makes broad task coverage possible.
- **Runtime isolation is a trade-off space.** The talk explicitly compares isolation technologies and their trade-offs — no single approach dominates, and the right choice depends on workload characteristics and safety requirements.
- **Persistence and storage are the next unlock.** The snippet identifies storage persistence as "the next major unlock for agent capabilities" — agents that accumulate and reference state across tasks represent a qualitatively different tier of usefulness.
- **Fleet orchestration is the harder problem.** Designing a single isolated sandbox is manageable; orchestrating and scaling a fleet of them introduces coordination, scheduling, and lifecycle management challenges that are distinct from single-sandbox design.
