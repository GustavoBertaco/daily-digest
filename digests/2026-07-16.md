## AWS What's New

**[AWS Lambda announces self-managed code storage](https://aws.amazon.com/about-aws/whats-new/2026/07/lambda-self-managed-code-storage/)**
AWS Lambda now lets functions reference source code directly from customer-owned S3 buckets, removing the intermediate copy Lambda previously created. This eliminates Lambda's code storage limits and reduces function activation time after creates and updates by skipping the copy step.

**[AWS Elastic Disaster Recovery now supports Amazon EBS volume initialization rate](https://aws.amazon.com/about-aws/whats-new/2026/07/aws-drs-fast-hydration/)**
AWS DRS now supports setting an EBS volume initialization rate on DRS-managed EC2 launch templates, allowing recovered volumes to reach full I/O performance faster during drills and failovers. The setting is applied automatically, reducing the window of degraded I/O while background S3 hydration completes.

**[AWS Lambda console provides a one-click setup prompt for coding agents](https://aws.amazon.com/about-aws/whats-new/2026/07/aws-lambda-prompt-coding-agents/)**
The Lambda console now offers a one-click prompt that configures a coding agent with AWS Serverless skills and the Serverless MCP server, embedding serverless best practices from the start. The setup is surfaced at every entry point in the Lambda console — onboarding, capability exploration, and first-function creation.

**[AWS IAM Identity Center achieves FedRAMP Class C Certification](https://aws.amazon.com/about-aws/whats-new/2026/07/aws-identity-center-fedramp/)**
AWS IAM Identity Center is now in scope for FedRAMP Class C across US East (Ohio), US East (N. Virginia), US West (N. California), and US West (Oregon). Organizations can now use IAM Identity Center for workforce access to AWS accounts and applications subject to FedRAMP Class C compliance requirements.

**[Introducing Amazon GuardDuty AI Protection for AWS AI workloads](https://aws.amazon.com/about-aws/whats-new/2026/07/amazon-guardduty-ai-protection-aws/)**
Amazon GuardDuty now extends threat detection to AI workloads on Amazon Bedrock and Amazon SageMaker, monitoring for anomalous model invocations, cost harvesting attacks, and prompt injection attempts. Security teams gain continuous, automatic coverage of AI-specific threats without manual configuration.

**[Amazon Managed Service for Apache Flink now offers AI Agent Skills to simplify building and operating Flink applications](https://aws.amazon.com/about-aws/whats-new/2026/07/amazon-managed-service-flink-agent-skills/)**
Amazon Managed Service for Apache Flink now provides AI Agent Skills that give coding assistants expert guidance for creating, troubleshooting, scaling, monitoring, networking, and cost-optimizing Flink applications. The skills are designed to keep streaming workloads healthy and accelerate development of new Flink applications.

**[AWS Security Hub now provides AI inventory for organization-wide visibility of AI assets](https://aws.amazon.com/about-aws/whats-new/2026/07/aws-security-hub-ai/)**
AWS Security Hub now delivers a continuously updated, organization-wide inventory of AI assets — agents, models, and pipelines — alongside their security posture. Central security teams can connect these assets to active threats and misconfigurations, closing the visibility gap that grows as AI workloads proliferate.

**[Amazon WorkSpaces Personal simplifies bulk PCoIP to DCV protocol migration](https://aws.amazon.com/about-aws/whats-new/2026/07/amazon-workspaces-personal-pcoip/)**
Amazon WorkSpaces Personal now supports automated rollback and migration of stopped WorkSpaces from PCoIP to DCV, building on the earlier console-based migration workflow and checkpoint snapshot support. **[Amazon Aurora DSQL is now available in Europe (Spain)](https://aws.amazon.com/about-aws/whats-new/2026/07/amazon-aurora-dsql-available-in-spain/)**
Amazon Aurora DSQL is now available as a single-Region cluster in the Europe (Spain) Region. Aurora DSQL is a serverless, distributed SQL database with active-active high availability and multi-Region strong consistency, expanding its regional footprint with this launch.

**[AWS Storage Gateway adds console support for copying file shares across gateways](https://aws.amazon.com/about-aws/whats-new/2026/07/aws-storage-gateway-console-copy-file-shares/)**

## AI Engineer

**[WTF Is the Context Layer? The Missing Infrastructure for Production Agents — Prukalpa Sankar](https://www.youtube.com/watch?v=8G_1-3IO4ZQ)**
Production AI agents are failing not because models are too weak but because they lack the organizational context that makes human experts valuable — business definitions, procedural knowledge, and operational norms that don't appear in training data. Drawing on hundreds of production deployments, Prukalpa Sankar argues that context is missing infrastructure, not a per-deployment tuning problem.

- **Models are not the bottleneck.** Top models have moved from failing the bar exam to scoring in the top 1% in two years, yet the same agents still can't answer simple business questions that any domain expert handles instantly.
- **The demo-to-production gap is a context gap.** Real business questions require precise terminology, nuanced operational norms, and business definitions that aren't in the model's training data.
- **Context is not documentation.** Business definitions and procedural knowledge are embedded in human practice, not static docs — the "context layer" is the infrastructure required to capture and serve them to agents at query time.
- **Abandonment is the real failure mode.** The pattern across hundreds of deployments is not a dramatic crash but a business team quietly stopping use within a month of going live.
- **The scale of the evidence makes this systemic.** Hundreds of production deployments pointing to the same root cause frames context as a structural gap in the AI stack, not an edge case.

**[Don't Ship Skills Without Evals — Philipp Schmid, Google DeepMind](https://www.youtube.com/watch?v=0vphxNt4wyk)**
Agent skills — the discrete capabilities that make agentic systems work — are being shipped at scale with no systematic evaluation, relying instead on ad hoc manual checks that would never pass code review. Philipp Schmid presents the full lifecycle of building reliable agent skills, from definition through triggering to a lightweight eval harness.

- **The analogy is direct: skills are code.** Teams that wouldn't merge code without tests are routinely shipping skills after two manual runs and a colleague's thumbs-up — the talk frames this as an industry-wide practice gap.
- **Triggering correctly is the first test to write.** A skill that doesn't fire when it should, or fires when it shouldn't, is broken regardless of what happens in the execution path; this is the most commonly skipped check.
- **A lightweight eval harness is achievable.** The talk covers building an evaluation framework that catches failures before users encounter them, without requiring a full ML testing infrastructure.
- **The gap is pervasive, not isolated.** Thousands of agent skills exist across the industry; almost none are tested systematically — making this a structural risk as agent skill libraries grow.

**[Forward Deployed Engineering at Cursor — Pauline Brunet](https://www.youtube.com/watch?v=APqXGyCoGW4)**
Pauline Brunet, VP of Forward Deployed Engineering at Cursor, describes how Cursor has built a team and operating model designed to embed alongside enterprise customers, configure Cursor for their environments, and co-build alongside their software and transformation teams to drive measurable returns from AI-assisted development.

- **Forward deployed is not sales engineering.** The function co-builds and configures Cursor alongside customer software teams — going significantly deeper than a typical implementation or enablement engagement.
- **Adoption and ROI are the explicit mandate.** The team's stated mission is to drive "meaningful returns," not just onboard customers — implying outcome measurement is built into the operating model.
- **The motion had to be invented.** Brunet describes building both the team and the playbook, signaling that forward deployed engineering at AI-native companies is still an emerging practice without a standard blueprint.

**["The engineer of the future is the person who is able to choose what is worth doing." — Addy Osmani](https://www.youtube.com/watch?v=n97BCfyFIvw)**
Addy Osmani's closing keynote argues that the defining skill for software engineers is no longer writing code but choosing what is worth building — as AI agents automate more of code generation, accountability, judgment, and system ownership become the durable differentiators.

- **The role is rebundling, not disappearing.** Engineers are absorbing adjacent responsibilities — system ownership, architectural judgment, accountability for outcomes — as code generation is increasingly automated.
- **Choosing what to build is the scarce skill.** The bottleneck in the AI agent era shifts from "can we build this?" to "should we build this, and what is it worth doing?" — a judgment call that doesn't automate.
- **Harnesses and loop engineering are emerging primitives.** The keynote introduces vocabulary — "harnesses" and "loop engineering" — for the practices that govern AI-assisted development pipelines, pointing to a new engineering discipline forming around agent oversight.
- **Human accountability increases with automation.** The throughline is that as production automation increases, the humans responsible for outcomes become more consequential, not less.

**[The Great Loops Debate — Dex Horthy, Geoff Huntley, Ian Livingstone, Greg Pstrucha, @insecure-agents](https://www.youtube.com/watch?v=c35YoMdnI78)**
An Oxford-style structured debate on whether the current wave of agentic loops deliver in production or whether a meaningful hype-to-reality gap remains — with credible engineers arguing both sides under debate rules that require affirmative positions.

- **Oxford format produces sharper claims than panels.** Both sides must make affirmative arguments rather than hedging; Team No Delta claims loops are already a silver bullet and represent a genuine step up the autonomy curve toward software factories.
- **Team Delta holds the gap is real.** The counter-position is that production outcomes lag behind community expectations for loop-based agents — the hype is running ahead of what reliably works today.
- **The debate itself is a signal.** Credible engineers disagree sharply on whether loops are production-ready, indicating that the evidence base is either thin or unevenly distributed across use cases and teams.
- **"Software factories" is the benchmark being contested.** Team No Delta frames loops as a step toward autonomous software factories; Team Delta's implicit challenge is that this framing describes a trajectory, not a current reality.
