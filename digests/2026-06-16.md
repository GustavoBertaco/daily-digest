# 🤖 AI & Machine Learning

## AI Engineer

**[Why MCP and ChatGPT Apps Use Double Iframes — Frédéric Barthelet, Alpic](https://www.youtube.com/watch?v=c-2eEv2ou7Y)**
Frédéric Barthelet of Alpic traces exactly why ChatGPT's MCP app rendering lands on a double-iframe architecture — each simpler alternative fails for a concrete security or isolation reason, leaving the double iframe as the only viable option.

- **`srcdoc` is ruled out first.** Inline content rendered via `srcdoc` inherits the parent origin, so ChatGPT's existing Content Security Policy blocks all third-party scripts the MCP app might need.
- **Relaxing the CSP creates a worse problem.** If the CSP is loosened to let the app run, any MCP app can read ChatGPT's own `localStorage` and cookies — an unacceptable privilege escalation.
- **`sandbox` closes the storage hole but opens another.** Adding a `sandbox` attribute removes origin-keyed storage access, which fixes the leakage problem but breaks apps that legitimately need storage.
- **`allow-same-origin` is a known sandbox escape.** Restoring storage access by adding `allow-same-origin` back to the sandbox is the classic documented escape from the `sandbox` attribute — it effectively cancels the protection.
- **The double iframe is what survives elimination.** An outer sandboxed iframe severs the origin relationship with ChatGPT; an inner iframe inside it can then be granted `allow-same-origin` relative to its own distinct origin without any privilege path back to the parent host.
