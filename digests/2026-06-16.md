# 🤖 AI & Machine Learning

## AI Engineer

**[Why MCP and ChatGPT Apps Use Double Iframes — Frédéric Barthelet, Alpic](https://www.youtube.com/watch?v=c-2eEv2ou7Y)**
Frédéric Barthelet walks through the security constraints that forced ChatGPT's MCP app renderer into a double-iframe architecture — each simpler alternative either breaks third-party script execution or opens a meaningful attack surface.

- **`srcdoc` is a non-starter due to origin sharing.** An iframe using `srcdoc` inherits the parent's origin, which means ChatGPT's Content Security Policy blocks all third-party scripts that MCP apps depend on.
- **Relaxing the CSP creates a worse problem.** Dropping the CSP restriction to unblock those scripts would allow any embedded app to read ChatGPT's `localStorage` and cookies — a direct session hijack vector.
- **`sandbox` removes the attack surface but breaks storage.** Adding the `sandbox` attribute neutralizes origin-scoped storage, which breaks apps that rely on `localStorage` for state.
- **`allow-same-origin` is a classic sandbox escape.** Restoring `allow-same-origin` inside a sandboxed iframe defeats the sandbox entirely, granting full origin access again — the exact threat it was meant to prevent.
- **The double iframe is what survives the elimination.** Nesting an iframe inside another iframe is the remaining option after every simpler path is ruled out on security grounds; it isolates the app's origin from ChatGPT's while preserving the storage behavior MCP apps need.
