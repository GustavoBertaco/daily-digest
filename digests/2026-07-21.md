## AI Engineer

**[Why Your Agent Disagrees With Itself (And What To Do About It) - Diane Lin, Datadog](https://www.youtube.com/watch?v=wEc9aG7cRQc)**
Diane Lin (Datadog) argues that LLM inconsistency — returning materially different answers to the same task — is not an inevitable property of probabilistic models but a diagnosable product flaw, with the consequences sharpest in high-stakes domains like cybersecurity where a flip between "malicious" and "benign" has direct operational impact.

- **Inconsistency is a product flaw, not a model property.** Dismissing flip-flops as the "stochastic nature of LLMs" is an engineering abdication; the inconsistency is located and addressable.
- **Failures cluster at the decision boundary.** Flip-flops occur in the gray zone where the model is genuinely uncertain — which means the fix is sharpening signal in that specific region, not swapping models wholesale.
- **Cybersecurity makes the stakes concrete.** A flip between classifying an alert as a malicious threat vs. a benign event is not a minor quality regression — it can lead to disastrous operational outcomes.
- **Customer trust is the direct casualty.** Inconsistency in high-stakes agent outputs destroys the user's ability to rely on the system, which is the foundation any agentic product must establish before anything else matters.

**[Don't Let the LLM Drive - Ornella Bahidika & Joel Allou, Microsoft](https://www.youtube.com/watch?v=m24UKZomm7k)**
Ornella Bahidika and Joel Allou (Microsoft) make the case that LLMs should handle generation, not control flow: in their voice tutor, all decisions about when the lesson ends, whether the user answered correctly, and which step comes next are owned by an external harness — the LLM just shows up and talks.

- **The LLM talks; the harness thinks.** Separating generation from decision-making prevents the canonical multi-step agent failures: declaring done too early, skipping a check, looping on a step, and silently dropping steps mid-procedure.
- **Prompting alone is insufficient for structured flows.** Getting a reliable multi-step agent requires layered discipline: prompt design, tool-use constraints, and structural harness enforcement — each layer closing gaps the previous one leaves.
- **Every engineer shipping multi-step flows has felt this.** The talk frames these failure modes as universal, not edge-case, positioning the harness pattern as the standard engineering response rather than an advanced optimization.
**[Your Voice Agent Doesn't Need a Frontier Model - Joel Allou & Ornella Bahidika, Microsoft](https://www.youtube.com/watch?v=fnLBmfsI_Fg)**
Joel Allou and Ornella Bahidika (Microsoft) argue that frontier models are the wrong choice for voice agents — not for cost reasons, but because a pause longer than a held breath registers as broken, making time-to-first-token the only metric that actually governs model selection.

- **Latency is the product, not the capability.** A big model that thinks for a second has already lost the room; in voice, the speed to first token determines whether the interaction feels alive or broken.
- **Benchmark score is the wrong optimization target.** The model selection decision is effectively made by the latency constraint — choose the fastest model that fits the requirement, then use scaffolding to close the intelligence gap.
- **Scaffolding is what makes small models feel smart.** The harness and surrounding infrastructure compensate for a smaller model's limitations; the system carries the intelligence, not the model alone.
- **The reason is latency, not cost.** This reframe matters: using a smaller model is not a cost-cutting compromise, it is the correct engineering decision driven by the physics of voice interaction.

**[Enterprise Agents Have a Structure Problem - Ishita Daga, Tesla](https://www.youtube.com/watch?v=B8l81jhvHbI)**
Ishita Daga (Tesla) diagnoses why enterprise agents that can generate SQL, invoke tools, and follow workflows still fail: they have no understanding of how the business actually defines its data, and the standard remedies — longer prompts, more RAG, bigger models — do not address that root cause.

- **Technical capability is not business understanding.** A model that can write SQL still fails when it does not know what "customer" or "revenue" means in a specific organization's data model; it guesses at raw schemas instead of reasoning over business concepts.
- **The standard remedies are symptoms-first interventions.** Longer prompts, more RAG, and larger models all operate on the output side of the problem; none builds the semantic infrastructure the agent is missing.
- **The real fix is a machine-readable metadata layer.** Semantic retrieval infrastructure lets agents reason over business concepts rather than guessing at column names and table relationships — a structural change, not a tuning one.
- **This is a data infrastructure problem, not a model problem.** The talk reframes enterprise agent failure as a semantic metadata gap — building the machine-readable metadata layer is a data engineering investment, not a prompting or model-tuning fix.

**[When Agents Meet Physical Data: The Other Physics of Agent Harnesses - Dmitry Petrov, DataChain](https://www.youtube.com/watch?v=bUJgirn4_yc)**
Dmitry Petrov (DataChain) argues that the agent design intuitions built up from code and business automation — where recomputing is cheap — are actively harmful when applied to physical data at scale, where a single pass over terabytes of dashcam video in S3 can cost thousands of dollars and run for hours or days.

- **Physical data has different physics.** The default agent pattern — loop, inspect, re-derive — works when recompute is cheap; it fails catastrophically when the compute substrate is large-scale physical data.
- **The familiar mental model becomes the most dangerous instinct.** "Just re-run it" is valid for software automation; when a first pass costs thousands of dollars and runs for days, that instinct is the worst thing an agent can act on.
- **The coding agent intuition does not transfer.** In code and business automation, verification means re-running code and state fits in context; neither holds for terabyte-scale data workloads.
- **Preserving intermediate state becomes a core requirement.** If re-derivation is prohibitively expensive, the architecture must treat intermediate results as durable artifacts — not as throwaway compute that can be re-triggered on demand.

**[Build the AI GTM Agent That Knows the Buyer - Dr. Sajjan Kanukolanu, Position2 (Position Squared)](https://www.youtube.com/watch?v=ltv-L5oMPIs)**
Dr. Sajjan Kanukolanu (Position2) argues that optimizing an AI GTM agent for the first conversation is the wrong frame — the competitive advantage lies in the buyer intelligence gathered and acted on before the buyer sends a single message.

- **The first message is the wrong moment to optimize for.** By the time a buyer opens a conversation, a well-built GTM system already knows who they are, what they are looking for, how likely they are to convert, and how to route them.
- **Pre-conversation intelligence is the differentiator.** The gap between AI GTM agents that convert and those that don't is not in the conversation itself — it is in what the system has done before the conversation starts.
- **Most teams are only building the conversation layer.** Focusing on "the right conversation, the right offer" at message-send time means missing the upstream buyer intelligence work that makes those conversations effective in the first place.
- **Routing and intent prediction happen before the chat.** These are the engineering investments that determine GTM agent performance — not prompt quality at conversation time.

**[Agents Need Receipts, Not More Tool Calls - Armanas Povilionis, Alithea Bio](https://www.youtube.com/watch?v=Q9ycQHbDdJs)**
Armanas Povilionis (Alithea Bio) introduces Froglet, an open-source protocol for agent-to-agent compute, built around the premise that when agents buy, sell, and compose work across hosts, API logs and dashboards are not sufficient provenance — signed receipts are.

- **Logs and dashboards don't prove what happened between agents.** As agent networks become multi-hop and cross-host, the audit model must move from passive observation to cryptographic proof embedded in the protocol.
- **Froglet standardizes the agent transaction lifecycle.** A single signed flow — Descriptor to Offer to Quote to Deal to Receipt — handles named services, data-backed services, and open-ended compute under one surface.
- **The receipt is the accountability primitive.** Rather than relying on tool-call history, the protocol generates a signed artifact at each transaction that proves what was requested, agreed to, and delivered.
- **Agent-to-agent commerce needs a trust model.** When one agent discovers and invokes another across host boundaries, the trust question is non-trivial; Froglet encodes the answer in the protocol structure itself.

**[Agent Output Is Not UX: Rendering Layer Your LLM Pipeline Is Missing - Bala Ramdoss, Amazon Lens](https://www.youtube.com/watch?v=maTp79FD9gI)**
Bala Ramdoss (Amazon Lens) argues there is an overlooked rendering layer between model output and product experience — and that failing to engineer it is what kills agentic CX features that otherwise work in demo, a lesson grounded in deployments at hundreds of millions of mobile devices.

- **Model output quality and UX quality are separate problems.** An agent that produces the right answer can still fail to ship if the layer that transforms that output into something people actually use is not engineered.
- **The wall is hit at production, not in demo.** The feature works, the demo impresses, and then production UX degrades — Ramdoss locates the failure at the rendering layer, not the model or pipeline.
- **Scale surfaces what demos hide.** Operating at hundreds of millions of mobile devices creates rendering-layer pressures that no demo environment exposes.
- **The missing layer sits between pipeline and product.** Teams that focus only on model quality and tool call correctness skip the transformation step that makes model output usable in a real product interface.

**[Voice Agents That Handle Interrupts - Chintan Agrawal and Daniel Wirjo, AWS](https://www.youtube.com/watch?v=hMlLw1LeIK8)**
Chintan Agrawal and Daniel Wirjo (AWS) address the engineering gap between an impressive voice demo and a voice agent people actually want to talk to — where the response budget is 200 milliseconds (not seconds) and a wrong turn means a hang-up, not a retry — building a production agent with Pipecat, an open-source real-time AI framework.

- **Voice's error mode is permanent.** Unlike chat, where users retry, a voice agent that stumbles causes users to hang up; resilience is a harder requirement than in text-based agent interactions.
- **200 milliseconds is the entire response budget.** Not seconds — this margin collapses the acceptable engineering tolerance and forces different decisions at every component of the stack.
- **Barge-in, turn-taking, and silence detection are the hard problems.** The gap between voice demo and production lives in these three mechanics, not in model quality; they are also the three most commonly mishandled.
- **Pipecat is the framework used.** An open-source real-time AI framework that surfaces the production voice engineering challenges the workshop addresses directly.

**[Skills are the New SDKs - Elvin Aghammadzada, DataRobot](https://www.youtube.com/watch?v=LC3-P7v3yoI)**
Elvin Aghammadzada (DataRobot) argues that REST APIs, SDKs, and MCP tool-calling are all insufficient for coding agents — which still invent steps and break in production — because what is missing is a skill layer: versioned, task-specific packages that encode the platform team's workflow knowledge in a form agents can load and execute.

- **APIs are not teachable.** A platform with a REST API, an SDK, and MCP support still fails when a coding agent uses it, because the interface encodes what is callable, not how the platform is actually meant to be operated.
- **Agents invent steps when knowledge is absent.** Without explicit workflow encoding, agents hallucinate intermediate steps from general training, producing code that demos but breaks under production conditions.
- **The fix is a versioned skill layer.** Task-specific packages that encode step ordering, failure modes, and security hygiene give agents what they need — not just callable endpoints but executable knowledge about the platform.
- **Platform teachability is a new engineering discipline.** Building a skill layer is as real a product investment as the API itself, and the talk frames it as one of the hardest challenges teams encounter when trying to make their platforms agent-ready.
