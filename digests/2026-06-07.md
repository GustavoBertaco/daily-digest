## 📰 Netflix Tech Blog

### [Dynamic Repartitioning for Time Series Workloads](https://netflixtechblog.com/dynamic-repartitioning-for-time-series-workloads)
*Published: June 3, 2026*

- Netflix's TimeSeries Abstraction ingests petabytes of temporal event data with millisecond latency on top of Apache Cassandra 4.x — chosen for throughput, cost efficiency, and operational maturity at their scale
- The core problem: static partition schemas work at launch but become performance bottlenecks as data volumes grow or shrink unevenly across time ranges
- Dynamic repartitioning lets them rebalance Cassandra partitions without downtime — a non-trivial engineering challenge given Cassandra's immutable partition key design
- Key insight: treating repartitioning as a continuous background process rather than a one-time migration dramatically reduces operational risk

## 📰 Airbnb Tech

### [Sitar-agent: Building a reliable dynamic configuration sidecar at scale](https://medium.com/airbnb-engineering/sitar-agent)
*Published: June 4, 2026*

- Airbnb's config system Sitar commits several changes per minute — the hard problem isn't storing configs, it's getting them to thousands of service instances reliably and fast
- They built a Kubernetes sidecar (Sitar-agent) that lives alongside each pod and handles config delivery locally, removing the need for each service to maintain its own config polling logic
- Sidecar pattern means config delivery failures are isolated per pod rather than cascading — a config service outage no longer takes down the application
- Latency for config propagation went from seconds to sub-100ms across the fleet

### [When history fails you, borrow from geography](https://medium.com/airbnb-engineering/when-history-fails-you-borrow-from-geography)
*Published: June 2, 2026*

- Cold-start problem in ML recommendations: new listings have no history, so standard collaborative filtering can't rank them
- Airbnb's solution: substitute geographic proximity for temporal history — listings near similar properties inherit their performance signals until enough real data accumulates
- The approach generalises: any ML system that relies on historical data can use spatial/categorical neighbors as a warm-start proxy when history is sparse

## 🌐 Uber Engineering

### [Zero-Growth Stack: How Stack Allocation Can Save 10% CPU in Go](https://eng.uber.com/zero-growth-stack/)

- At Uber, Go runs ~65% of all services across 2M+ cores — a 1% CPU efficiency gain translates to millions of dollars, making micro-optimizations economically justified at this scale
- The fix: replace heap-allocated buffers in hot paths with stack-allocated equivalents, eliminating GC pressure and improving cache locality simultaneously
- 10% CPU reduction in a targeted service; Uber is now running the analysis fleet-wide to find other candidates — the tooling to identify high-GC-pressure hot paths is the real deliverable

### [Building Uber's Multi-Cloud Secrets Management Platform](https://eng.uber.com/multi-cloud-secrets/)

- Uber manages 150,000+ secrets across 5,000 microservices, 400 third-party vendors, and 400 SaaS integrations — at that scale, a broken secret rotation is an incident
- Moved from a single-cloud vault to a multi-cloud secrets platform that can survive a provider-level outage without cascading auth failures across the fleet
- Key design decision: secrets are versioned and rotated atomically — consumers always see a consistent view even mid-rotation

## 📺 AI Engineer

### [LLM Observability, Evaluation, Experimentation Platform — Dat Ngo, Arize](https://www.youtube.com/watch?v=RS3PFeMB5BU)
*Published: June 7, 2026*

- Core framing: when your system is nondeterministic and execution paths change every run, traditional observability (logs + metrics) is insufficient — you need telemetry that captures *what the agent decided*, not just what it executed
- Five eval signal types: LLM-as-judge, human feedback, golden datasets, deterministic checks, and business metrics — each catches different failure modes; using only one leaves blind spots
- Practical architecture: run evals at multiple scopes (single span, multi-turn, session-level) because tool-ordering bugs only surface at multi-turn scope, not per-span
- Key takeaway: if your agent called tool B before tool A and B depends on A, only telemetry catches it — code reviews and unit tests cannot audit agent decision sequences

### [Under 5 Minutes to a Deployed LLM Endpoint — Audry Hsu, RunPod](https://www.youtube.com/watch?v=RS3PFeMB5BU)
*Published: June 7, 2026*

- RunPod went from two failed crypto mining rigs on Reddit to $120M ARR and 500K developers — the origin story matters because their infrastructure is purpose-built by people who started as GPU renters themselves
- Demo: pick model from Hub, configure context window, deploy serverless on H100s — first request cold-starts in 41 seconds while the container and model load; subsequent requests are instant
- Pricing model is per-second of compute, not per-request — this matters for bursty workloads where per-request pricing punishes you during warm-up
- Current limitation: 41-second cold start is acceptable for async batch workloads but still too slow for synchronous user-facing calls

### [From MCP to Scale: Pipelines That Build Themselves — Rafael Levi, Bright Data](https://www.youtube.com/watch?v=...)
*Published: June 7, 2026*

- The hard part of web scraping has shifted from extraction to maintenance — sites change, scrapers break, and keeping them working at scale requires constant human intervention
- Bright Data's MCP-based approach: an agent inspects the site structure, generates a production scraper, and automatically repairs it when the site changes — the pipeline is self-maintaining
- Token efficiency trick: once a reusable parser is built for a site, switch from full-page parsing to the cached script — dramatically reduces tokens per extraction on subsequent runs
- This is the pattern that makes agentic data pipelines economically viable at scale: agent effort upfront, deterministic execution after

### [Evals Are Broken, Use Them Anyway — Ara Khan, Cline](https://www.youtube.com/watch?v=...)
*Published: June 6, 2026*

- Cline improved from 43% to significantly higher on Terminal Bench not by switching models but by tuning container CPU/memory, timeouts, and prompt engineering specific to Anthropic model families
- The uncomfortable truth: benchmark improvements often don't transfer across model families — a prompt tuned for Claude doesn't help you on Codex or Gemini
- Practical eval framework: after a run, send failures to a second agent that classifies them — you get a portfolio view of your failure modes instead of a single pass/fail number
- The meta-lesson: vibes aren't a system and benchmarks aren't gospel; the useful truth is somewhere in between, and you need structured failure analysis to find it

### [Text Diffusion — Brendon Dillon, Google DeepMind](https://www.youtube.com/watch?v=...)
*Published: June 4, 2026*

- Autoregressive models (GPT, Claude) generate left-to-right and cannot revise earlier tokens once generated — text diffusion uses bidirectional attention so the model can see future tokens and correct earlier mistakes mid-generation
- Demo result: on a math problem, Gemini 2.5 Flash answered 42 and stuck to it even after reasoning through it incorrectly; the Gemini Diffusion model answered 60, then 49, then corrected itself to 39 — the correction is only possible because it can revise
- Speed advantage: 24 denoising steps to generate 256 tokens is meaningfully faster than 256 autoregressive steps at scale
- Why this matters: text diffusion could be the architecture that finally decouples generation quality from generation length constraints
